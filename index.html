PD Controller for Autonomous Drive

A proportional-derivative control loop providing smooth heading correction and precise straight-line driving for first-generation VEX IQ robots. Written in C++ for VEXcode IQ.

Concept

What is a PD Controller?

A PD controller is a feedback loop that continuously measures the gap between where the robot is and where it should be — the error — and computes a corrective output each cycle. Two terms work together to do this:

velocity = (error × Kp) + ((error − prevError) × Kd)

Proportional term — Kp

Scales the correction proportionally to the current error. A 10° heading deviation produces twice the correction of a 5° deviation. The robot always pushes toward its target in direct proportion to how far off course it is.

Derivative term — Kd

Looks at the rate of change of error between cycles. When the robot is converging fast on its target, Kd applies a braking force to prevent overshoot. This is what transforms oscillation into a smooth, settled approach.

Why not PID?

Omitting the integral term eliminates integral windup — where accumulated error causes overcorrection after a long drive. On smooth VEX IQ surfaces, PD is sufficient and considerably easier to tune.

Setup — Step 2 — Critical

Verifying Motor Direction Before Anything Else

Before writing a single line of autonomous code, confirm that commanding each motor forward actually moves the robot forward. A reversed motor is indistinguishable from a miscalibrated PD gain during tuning — you will spend hours chasing a problem that does not exist.

Do this first

Do not skip this step. All calibration in the next stages assumes both motors are already oriented correctly.

The test procedure

  1. Decide which end is the front

    Choose one end of the robot and commit to it. Everything else depends on this definition.

  2. Test the right motor in isolation

    Write a minimal program that spins only the right motor forward at 50% for one second. With the robot's front facing away from you, the right wheel should turn such that the robot would travel forward.

  3. Test the left motor in isolation

    Repeat for the left motor. The left wheel should also drive the robot forward when commanded forward.

  4. Reverse a motor in code if needed

    Pass true as the second constructor argument. The left motor in this project is already reversed due to its mirrored mounting orientation.

C++ / VEXcode IQ
motor rightMotor = motor(PORT8, false);  // not reversed
motor leftMotor  = motor(PORT7, true);   // reversed — mirrored mount

Calibration — Step 3

Tuning the Proportional Gain, Kp

Kp is the first and only constant you touch in this stage. Set Kd to zero and leave it there until Kp is stable. Tuning both simultaneously makes it impossible to isolate which term is causing a given behavior.

What Kp does

Every loop, the controller multiplies the current heading error in degrees by Kp to produce a corrective velocity. If Kp is 1.0 and the robot is 5° off course, it applies a 5% velocity differential between the two motors. At Kp 3.0, the same 5° error produces a 15% differential.

Starting procedure

  1. Set Kd to zero

    Isolate Kp completely. Only the proportional term should be active.

  2. Start at Kp = 0.5

    Run a 1-meter straight drive. Observe whether the robot corrects slowly, promptly, or not at all.

  3. Raise in increments of 0.5

    After each increment, run the test again. Stop when the robot corrects promptly but has not started oscillating.

  4. Lock in just below oscillation

    Find the exact point where oscillation begins, step back by 0.5, and record this value. You will not change it again.

Diagnosing Kp behavior

Kp too low — robot drifts, correction is slow

Robot creeps toward target heading but never arrives cleanly. Raise Kp.

Kp correct — clean approach, heading holds

Robot converges firmly and holds the heading. This is the target behavior.

Kp too high — robot oscillates around heading

Robot swings past heading and overcorrects repeatedly. Step back, then add Kd.

Common mistake

Raising Kp past the oscillation threshold and concluding the controller is broken. Oscillation at high Kp is expected — it means you have gone too far. Step back and move on to Kd tuning.

C++ — Proportional only, Kd = 0
float error      = targetAngle - gyro.angle();
float correction = error * Kp;    // tune this first; Kd = 0

rightMotor.spin(fwd, speed + correction, pct);
leftMotor.spin(fwd,  speed - correction, pct);
Calibration — Step 4

Tuning the Derivative Gain, Kd

Once Kp is locked in, Kd handles the remaining oscillation by damping the controller's response as error collapses. Always tune Kd second.

What Kd does

Kd multiplies the change in error between the current and previous loop cycle. When the robot is converging rapidly, error is falling fast — Kd produces a braking output that opposes the overshoot before it happens. Think of it as a shock absorber that resists sudden change.

Starting procedure

  1. Restore the Kp you locked in

    Use the value from the previous stage, at or just below where oscillation appeared.

  2. Start at Kd = 0.05

    Run the same straight-drive test. Oscillation should reduce noticeably even at this small value.

  3. Raise in increments of 0.05–0.1

    Keep raising until oscillation disappears and the robot settles cleanly on its heading.

  4. Stop before the robot becomes sluggish

    Over-damping is when Kd is so large that derivative braking overpowers the proportional drive. Reduce slightly until both responsiveness and stability are good.

Diagnosing Kd behavior

Kd too low — oscillation persists

Not enough damping. Continue raising Kd incrementally.

Kd correct — fast, clean settling

Robot approaches and holds the heading with no oscillation. Optimal.

Kd too high — over-damped, sluggish response

Robot barely reacts to heading changes. Derivative braking is overwhelming proportional drive.

Common mistake

Using Kd to compensate for a Kp that is too high. If oscillation is severe, reduce Kp first, then add Kd. Kd cannot fully counteract a wildly oversized Kp.

C++ — Full PD loop
float error      = targetAngle - gyro.angle();
float derivative = error - previousError;
float correction = (error * Kp) + (derivative * Kd);

rightMotor.spin(fwd, speed + correction, pct);
leftMotor.spin(fwd,  speed - correction, pct);

previousError = error;
Reference

Function Reference

void pivotRightToAngle(float targetAngle)

Pivots around the stationary right wheel until the gyroscope reads targetAngle. The left wheel drives; the right wheel holds.

void pivotLeftToAngle(float targetAngle)

Pivots around the stationary left wheel. The right wheel drives; the left wheel holds. Produces a tighter inside radius on left turns.

void spinToAngle(float targetAngle)

Point turn — both wheels counter-rotate simultaneously. Fastest heading change; robot pivots around its own center.

void driveStraight(float speed, float targetDistanceDegrees, float targetAngle)

Drives both motors at speed while applying PD heading correction against targetAngle. Exits once drive motors have traveled targetDistanceDegrees of rotation for 5 consecutive cycles.

Reference — Implementation

Implementation Details

Differential correction in driveStraight

Both motors are driven at a constant base speed. Each loop, the heading correction is applied in opposite directions. If the robot drifts right, the right motor speeds up and the left slows down, nudging the robot back on course. 

Concept
rightVelocity = speed + headingCorrection
leftVelocity  = speed - headingCorrection

Angle normalization

All angular errors are wrapped to −180° to +180°. Without this, a robot at 350° heading for 10° might compute a 340° error — the long way — instead of the correct 20° short path. 

C++
while (error >  180) error -= 360;
while (error < -180) error += 360;

Settled count

Every function requires the robot to remain within the tolerance band for 5 consecutive loop cycles before stopping. This prevents a single noisy gyroscope reading from triggering a false exit.

Motor settings

All velocities are clamped to [-100, +100]%. All motors use hold braking mode, which actively resists movement after stopping — important for maintaining position between autonomous actions.

Startup sequence

  1. TouchLED turns red

    Calibration in progress. Do not move the robot.

  2. Gyroscope slow calibration

    calSlow gives the gyroscope more time to self-zero. Robot must remain completely still.

  3. Gyroscope resets to 0°

    Current physical orientation becomes the zero reference for all heading calculations.

  4. TouchLED turns green

    All systems calibrated. Place robot in starting position before pressing.

  5. Press TouchLED to start

    The autonomous routine inside route() begins executing.

Usage

Getting Started

  1. Open VEXcode IQ and create a new C++ project

    File → New Project → C++ Template.

  2. Source code — restricted access

    The full C++ implementation is private. Access is granted exclusively to M&P Robotix coaches. Please do not attempt to request or obtain the source outside of that context.

  3. Verify motor directions

    Follow the Motor Direction Check section above. This step is not optional.

  4. Tune Kp, then Kd

    Follow the calibration sections in order. Rushing this stage produces unreliable autonomous runs.

  5. Write your autonomous route

    Add your sequence of calls inside the route() function.

  6. Download, place robot, wait for green, press TouchLED

    The green light confirms calibration is complete. Only press once the robot is in its starting position.
